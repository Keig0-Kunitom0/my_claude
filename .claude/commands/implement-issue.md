description: スペック駆動開発でissueの実装を自律的に行う
---

## 概要
指定されたissueについて、要件定義 → 設計 → タスク分解 → 実装までを、
ユーザーと合意したドキュメント（requirements / design / tasklist）を唯一の正として自律的に進める。
レビューやPR作成はこのコマンドの範囲外とし、実装の完遂を目的とする。

重い読み込み（影響範囲調査・実装）はサブエージェントに委譲し、このコマンド自身はオーケストレーションに専念することで、
コンテキストウィンドウの消費を抑える。

## 前提・設定
`git` と GitHub CLI (`gh`) が使える環境で動作する。実行開始時に以下を各自の環境へ解決すること。

- **対象リポジトリ**: カレントディレクトリのリポジトリ（`gh repo view` で確認）
- **デフォルトブランチ**: `gh repo view --json defaultBranchRef -q .defaultBranchRef.name`（以下 `<base>`）
- **ステアリング格納先**: リポジトリルートの `.steering/<issue番号>-<タイトルのスラッグ>/`（無ければ作成）
- **テスト / Lint コマンド**: `CLAUDE.md` やプロジェクト設定（package.json / Makefile 等）から判定する。
  特定できない場合はユーザーに確認する。以降、サブエージェントにもこのコマンドを渡す。

`$ARGUMENTS` には issue番号・issueのURL・補足ドキュメントのパスのいずれか（複数可）を受け取る。

## ステップ1: issueの理解と作業ブランチの作成
1. `$ARGUMENTS` のissueを `gh issue view <issue番号> --comments` で読み込む（タイトル・本文・コメント全て）。
   ファイルパスが含まれる場合はその仕様書も読む。
2. `<base>` を最新化する（`git checkout <base> && git pull`）。このコマンドではリモートへの push は行わない。
3. 作業ブランチを切る: `git checkout -b <issue番号>/<タイトルのスラッグ>`

### 軽量フロー
ステップ2〜4のステアリングドキュメント（requirements / design / tasklist）作成と承認ゲート、およびステップ5のサブエージェント委譲を省略し、以下の最小手順で完遂する。
1. 影響範囲（参照箇所・依存関係）を確認し、結果をユーザーに簡潔に共有する。
2. 対応方針をユーザーに確認し、合意を得る（`AskUserQuestion` 等の利用を推奨）。
3. 実装し、テスト / Lint を通したうえでローカル commit する。
4. 実装完了をユーザーに報告する（レビューやPR作成は別途）。

軽量フローでも **禁止事項** と **原則**（`git push` をしない、既存仕様を勝手に変えない、失敗を握り潰さない）は重量級フローと同様に厳守する。

## ステップ2: requirements.md の作成
1. `implement-issue-spec-docs` スキルに従い、`.steering/<...>/requirements.md` を作成する。
2. ユーザーにレビューを依頼し、**approveされるまで**ブラッシュアップする。approve後にステップ3へ進む。

## ステップ3: design.md の作成
1. 設計に着手する前に `codebase-investigator` サブエージェントを起動し、影響範囲と既存実装パターンの調査レポートを得る。
2. `implement-issue-spec-docs` スキルと調査レポートに従い、`.steering/<...>/design.md` を作成する。
3. ユーザーにレビューを依頼し、**approveされるまで**ブラッシュアップする。approve後にステップ4へ進む。

## ステップ4: tasklist.md の作成
1. `implement-issue-spec-docs` スキルに従い、`.steering/<...>/tasklist.md` を作成する（1コミット相当の粒度）。
2. requirements.md / design.md の要件を満たすかセルフチェックし、不足を補う。
3. ユーザーにレビューを依頼し、**approveされるまで**ブラッシュアップする。approve後にステップ5へ進む。

## ステップ5: 実装
カレントが作業ブランチであることを確認し、**tasklist.md のチェックボックスを上から順に1つずつ** `task-implementer` サブエージェントに委譲する。

各チェックボックスについて:
1. `task-implementer` を起動し、以下を渡す。
   - 対象チェックボックスの番号と内容
   - ステアリングディレクトリのパス（`.steering/<...>/`）
   - テスト / Lint コマンド
2. レポートを受け取り、結果に応じて進める。
   - **成功**: 次のチェックボックスへ進む。
   - **対応方針の確認要求**: レポートに記載された確認事項をユーザーに伝え、方針の指示を仰いでから再開する。
   - **中断（5回試行しても失敗）**: 失敗箇所・再現条件・原因をユーザーに伝え、指示を仰ぐ。
3. 全チェックボックス完了後、`<base>` との差分が requirements / design / tasklist を満たすかセルフチェックする。
   不足があればユーザーに確認のうえ修正し、必要に応じて3ドキュメントも更新する。
4. 実装完了をユーザーに報告する（レビューやPR作成は別途行う）。

## 禁止事項
- **`git push` をしない。** このコマンドが行うのはローカルでの commit までとする（リモート反映・PR作成は範囲外）。
- **テストを通すためだけに既存の仕様を変更しない。** 既存箇所の仕様変更が必要な場合は、勝手に変更せず必ずユーザーに対応方針を確認する。
- **失敗を握り潰さない。** 原因が特定できないまま、テストの skip や期待値の安易な書き換えなど回避的な手段でテストを通さない。

## 原則
- **スペック駆動を厳守する。** 3ドキュメントに書かれた内容のみを実装する。
  やむを得ず逸脱する場合は必ずユーザーに報告し approve を得る。
- **途中再開**: 同じ issue 番号を `$ARGUMENTS` に渡すと、対応する作業ブランチと `.steering/<...>/` を特定して再開できる。
  `tasklist.md` のチェック状態を正として進捗を判断し、中断していた場合は `blockers.md` を先に読んで文脈を復帰してから再開する。
- **判断と詰まりの記録**（自動圧縮で文脈が失われないための保険。`implement-issue-spec-docs` スキルのテンプレート参照）。
  - `decisions.md`: 重要な設計判断・技術選定を行ったとき、採用理由・代替案・トレードオフを追記する。
  - `blockers.md`: 実装が中断したとき、事象・再現条件・試したこと・原因・確認事項を追記する。
