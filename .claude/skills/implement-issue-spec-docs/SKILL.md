# スペック駆動ドキュメントの作成

ステアリングドキュメントはAIエージェントが向かう先を指し示す「北極星」であり、実装の唯一の正となる。
各ドキュメントは `templates/` の雛形をベースに、以下の基準で作成する。

## 共通の品質基準
- **断定的・定量的に書く**（「適切に」「検討する」などの曖昧語を避ける）。
- 構造化された技術要素（**コード例・メソッド名・Mermaid図**）を含めると、AIの指示追従性が上がる。
- ファイル名・行番号・クラス名など、参照先を具体的に書く。

## requirements.md — 何を・なぜ
テンプレート: `templates/requirements.md`
必須セクション:
- **背景**: なぜ必要か。解決する問題（問題の詳細・根本原因）。
- **変更・追加・削除する機能の説明**: 対象を表で列挙する（対象 / 場所 / 現状 / 変更後・用途）。
- **仕様**: 満たすべき振る舞いを断定的に箇条書きする。
- **スコープ外**: 今回やらないことと、その理由（スコープの肥大を防ぐ）。

## design.md — どう作るか
テンプレート: `templates/design.md`
必須セクション:
- **具体的な仕様・設計**: 現状の問題と解決方針。データフローや処理順は **Mermaid（sequence図 / ER図）** で図示する。設計詳細はコード例で示す。
- **実装アプローチ**: 手順（番号付き）、変更するコンポーネント（表）、データ構造の変更（なければ「なし」と明記）、影響範囲の分析。
- **動作確認対象**: 対象ごとに「○○で○○を確認する」と具体的な確認手順を書く。確認方法は画面操作・CLI実行・APIコール・ログ確認など対象に合わせ、可能な限り再現しやすい手順にする。

## tasklist.md — 具体的な実装タスク
テンプレート: `templates/tasklist.md`
ルール:
- チェックボックスは **1コミット相当の粒度**にする（例: 「○○メソッドの追加」「○○を○○に修正」）。
- 各チェックボックスの下に、対象ファイルと具体作業をネストする。**実装コード例**やテスト追加を含める。
- 作成後、requirements / design の要件を全て満たすかセルフチェックする。不足があればチェックボックスを追加する。
- 対応しないタスクは `→ **スキップ**（理由）` と明記する。

## decisions.md / blockers.md — 自走中の記憶（条件付き作成）
コンテキストウィンドウの自動圧縮で文脈が失われても、判断と詰まりを後から追えるようにするための記録。
**毎回作る必要はなく、該当する事象が起きたときにだけ追記する。**

- **decisions.md**（テンプレート: `templates/decisions.md`）: design.md に書かれていない設計判断・技術選定や、トレードオフのある選択を行ったとき、採用理由・代替案・影響とともに追記する。
- **blockers.md**（テンプレート: `templates/blockers.md`）: 実装が中断したとき、事象・再現条件・試したこと・原因・確認したいことを追記する。再開時は最初にこれを読む。
